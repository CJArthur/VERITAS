# VERITAS — Платформа верификации образовательных документов

Криптографически защищённая проверка подлинности дипломов, сертификатов и профессиональных лицензий. Работодатель сканирует QR-код — за 3 секунды получает математически доказуемый результат. Без звонков в ВУЗ. Без регистрации.

Разработано для [Diasoft](https://diasoft.ru) · 2026

## Концепция

**Проблема.** По данным ряда исследований, ежегодно в России продаётся сотни тысяч поддельных дипломов. Работодатели не проверяют образование кандидатов, потому что официальный запрос в ВУЗ занимает дни, государственный реестр ФРДО не покрывает частные учебные центры, а Госуслуги не предоставляют открытый API для бизнеса.

**Решение.** VERITAS — дополнительный слой криптографического доверия поверх существующей системы. Организация загружает документ → система вычисляет хэш и создаёт цифровую подпись → выпускник получает QR-код → проверяющий сканирует и мгновенно видит результат.

VERITAS **не заменяет** ФРДО и государственную систему. Это самостоятельный инструмент для организаций, которые хотят предоставить проверяющим криптографически верифицируемое подтверждение своих документов.

## Правовая основа

### Почему VERITAS законен

**Добровольная платформа (не государственный реестр).** VERITAS — частный сервис, в котором организации регистрируются добровольно. Участие не заменяет обязательную подачу данных в ФРДО (Федеральный реестр документов об образовании), а дополняет её. Работодатель использует результат как дополнительное подтверждение, наряду с просмотром оригинала документа.

**Нормативная база:**
- **ФЗ-273 «Об образовании в РФ», ст. 60** — образовательные организации несут ответственность за достоверность выданных документов. VERITAS помогает им исполнять это требование в цифровом виде.
- **ТК РФ, ст. 65** — при приёме на работу работодатель вправе потребовать документ об образовании и **убедиться в его подлинности**.
- **ГК РФ, ст. 421** — свобода договора: стороны вправе использовать любые частные инструменты верификации.
- **ФЗ-149 «Об информации», ст. 7** — общедоступная информация (факт выдачи диплома, специальность, год) может свободно распространяться при наличии согласия субъекта.
- **ФЗ-63 «Об электронной подписи»** — электронная подпись обеспечивает юридическую значимость документа. В продакшне подпись выполняется алгоритмами ГОСТ Р 34.10-2012, сертифицированными ФСБ.

### Как это работает юридически

1. **Студент или выпускник** привязывает диплом к аккаунту и сам создаёт ссылку для работодателя — тем самым даёт **явное согласие на передачу данных** конкретному лицу (ФЗ-152, ст. 9).
2. **Организация** регистрируется на платформе, предоставляет ОГРН и (для образовательных организаций) номера лицензии и аккредитации Рособрнадзора. Администратор платформы верифицирует эти данные.
3. **Работодатель** получает только те данные, которые студент разрешил показать: факт подлинности, специальность, год выпуска. ФИО и другие ПДн показываются только при наличии действующей ссылки от студента.

### Персональные данные и 152-ФЗ

| Требование | Реализация |
|---|---|
| Шифрование ПДн | AES-256-GCM на уровне приложения; в БД — только зашифрованный BYTEA |
| Согласие субъекта | Явное согласие при создании TTL-ссылки; без ссылки студента доступ к ПДн закрыт |
| Минимизация данных | Работодатель видит только то, что необходимо для верификации |
| Журналирование | Иммутабельный лог всех обращений к данным (PostgreSQL RULE) |
| Хостинг | Рекомендуется Selectel, Яндекс.Облако или КРОК (серверы в РФ) |
| Уровень защищённости | УЗ-4 (базовый для общедоступных ПДн) |

## Архитектура

Монорепозиторий. Backend и frontend разворачиваются независимо.

```
veritas/
├── backend/
│   ├── app/
│   │   ├── api/          # Роутеры (auth, public, student, university, admin, employer)
│   │   │   └── services/ # Бизнес-логика: крипто, импорт CSV, email, JWT
│   │   ├── db/           # ORM-модели, подключение к PostgreSQL и Redis
│   │   └── utils/        # JWT, куки, зависимости ролей
│   └── migrations/       # Alembic-миграции
│
└── frontend/
    ├── app/              # Next.js App Router: страницы по ролям
    │   ├── v/[token]/    # Публичная верификация (SSR, без авторизации)
    │   ├── student/      # Личный кабинет выпускника
    │   ├── university/   # Кабинет организации + аналитика
    │   └── admin/        # Панель администратора платформы
    └── components/       # UI-компоненты
```

### Ключевые архитектурные решения

**Криптографическая подпись.** При добавлении диплома вычисляется HMAC-хэш канонической строки (порядок полей фиксирован и версионирован) и создаётся подпись приватным ключом организации. При проверке подпись верифицируется публичным ключом: любое изменение данных в БД делает подпись недействительной.

**CryptoProvider — абстракция алгоритмов.** Одна строка конфига переключает реализацию:
```python
CRYPTO_PROVIDER=sha256_rsa   # хакатон (SHA-256 / RSA-2048)
CRYPTO_PROVIDER=gost         # прод (ГОСТ Р 34.11-2012 + ГОСТ Р 34.10-2012 через КриптоПро CSP)
```
Бизнес-логика не зависит от конкретного алгоритма.

**Иммутабельный журнал верификаций.** Таблица `verification_logs` защищена PostgreSQL RULE, запрещающим UPDATE и DELETE на уровне СУБД — не программным ограничением. История верификаций не может быть изменена или удалена даже при компрометации приложения.

**Privacy-first доступ.** Студент создаёт TTL-ссылку с явным указанием получателя, сроком действия и лимитом просмотров. Без ссылки студента публичный доступ к диплому закрыт. Студент видит, кто и когда смотрел его документ.

**Live revoke.** При отзыве документа организацией все открытые страницы верификации обновляются в реальном времени (polling каждые 6 секунд через отдельный лёгкий эндпоинт без ПДн). Студент не может «заморозить» страницу с действующим статусом до момента проверки.

**Автоматическая верификация организаций.** При рассмотрении заявки на регистрацию ВУЗа или учебного центра система автоматически проверяет ОГРН: сначала математически (контрольная сумма по алгоритму ФНС, без сети), затем — через DaData API (обёртка над реестром ФНС/ЕГРЮЛ), получая официальное наименование, ИНН и статус организации. Результат отображается в интерфейсе администратора в виде светофора с рекомендацией. Администратор нажимает одну кнопку — без открытия внешних сайтов вручную. Настраивается переменной `DADATA_API_KEY`.

**Открытый API для работодателей.** HR-системы и ATS проверяют документы программно через Bearer-ключ. Каждый запрос фиксируется с указанием организации-верификатора. Ключ хранится только в виде SHA-256-хэша.

**Сессии.** JWT в `httpOnly`-куках, недоступных JavaScript. Refresh-токены хранятся в Redis с ротацией при каждом обновлении и немедленной инвалидацией при logout.

## Роли и регистрация организаций

| Роль | Как создаётся | Возможности |
|---|---|---|
| `student` | Самостоятельная регистрация по email | Привязать диплом, создавать TTL-ссылки, скачивать QR, смотреть историю проверок |
| `university_staff` | Регистрация организации → ожидание одобрения | Управление документами, импорт CSV, аналитика верификаций |
| `super_admin` | Bootstrap через env (один раз) | Одобрение организаций, выдача API-ключей работодателям, общая статистика |

### Процесс регистрации организации

Регистрация — двухэтапная: автоматическая проверка данных + финальное решение администратора.

**Этап 1 — Заявка.**  
Представитель организации регистрируется на `/register/university` и указывает:
- Полное наименование организации
- ОГРН (13 цифр для юрлиц или 15 для ИП)
- Номер лицензии на образовательную деятельность (выдаётся Рособрнадзором; необязателен для сертификационных организаций)
- Номер государственной аккредитации (только для ВУЗов с государственными дипломами)

**Этап 2 — Автоматическая верификация.**  
При открытии заявки администратором система автоматически (без участия человека):

1. **Контрольная сумма ОГРН** — алгоритм ФНС, работает без сетевых запросов. Математически невозможно пройти с некорректным ОГРН.
2. **Поиск в ЕГРЮЛ через DaData API** — возвращает официальное наименование из реестра ФНС, ИНН и статус (`ACTIVE` / `LIQUIDATED` / `BANKRUPT`). Требует `DADATA_API_KEY` (бесплатный тариф: 10 000 запросов/мес).
3. **Прямые ссылки** — администратор видит кнопки «ФНС ЕГРЮЛ», «Лицензии Рособрнадзор», «Аккредитации» с готовыми URL для финальной ручной проверки лицензии.
4. **Рекомендация** — система выдаёт `approve` / `reject` / `manual_review` с обоснованием.

Администратор смотрит на светофор и нажимает «Одобрить» или «Отклонить». Ручной поиск в реестрах — не нужен.

**Конфигурация DaData:**
```env
DADATA_API_KEY=ваш_ключ_с_dadata.ru  # опционально; без ключа — только контрольная сумма
```

> Организации, выдающие сертификаты (Skillbox, корпоративные учебные центры), не обязаны иметь лицензию Рособрнадзора — достаточно ОГРН. Тип документа `certificate` или `professional_license` указывается явно при добавлении каждого документа.

## Локальный запуск

### Требования

- Python 3.11+
- Node.js 18+
- PostgreSQL 15
- Redis 7

### 1. Клонировать репозиторий

```bash
git clone <repo-url>
cd veritas
```

### 2. Backend

```bash
cd backend

# Создать виртуальное окружение
python -m venv .venv
source .venv/bin/activate        # macOS / Linux
# .venv\Scripts\activate         # Windows

# Установить зависимости
pip install -r requirements.txt
# Или через uv (быстрее):
# pip install uv && uv sync
```

Создать `backend/.env` из шаблона:

```bash
cp example.env .env
```

Минимально необходимые переменные:

```env
DB_HOST=localhost
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=your_password
DB_NAME=veritas

UPSTASH_REDIS_URL=redis://localhost:6379
UPSTASH_REDIS_TOKEN=          # оставить пустым для локального Redis

SECRET_KEY=                   # минимум 32 символа: openssl rand -hex 32

MAIL_SERVER=smtp.yandex.ru
MAIL_PORT=587
MAIL_USERNAME=your@email.com
MAIL_PASSWORD=your_smtp_password
MAIL_FROM=your@email.com
MAIL_FROM_NAME=VERITAS
MAIL_TLS=true
MAIL_SSL=false

FRONTEND_URL=http://localhost:3000
BASE_VERIFICATION_URL=http://localhost:8200/api/v1/verify
ALLOWED_ORIGINS=http://localhost:3000

BOOTSTRAP_SUPER_ADMIN_EMAIL=admin@yourdomain.com
```

```bash
# Применить миграции
PYTHONPATH=. alembic upgrade head

# Запустить сервер
PYTHONPATH=. uvicorn app.main:app --reload --port 8200
```

Swagger UI: `http://localhost:8200/docs`

### 3. Frontend

```bash
cd frontend
npm install
```

Создать `frontend/.env.local`:

```env
NEXT_PUBLIC_API_URL=http://localhost:8200
API_URL=http://localhost:8200
```

```bash
npm run dev
```

Приложение: `http://localhost:3000`

### 4. Создать первого администратора

1. Задать `BOOTSTRAP_SUPER_ADMIN_EMAIL` в `backend/.env`
2. Перейти на `http://localhost:3000/register` и зарегистрироваться с этим email
3. Подтвердить email — аккаунт получит роль `super_admin`
4. Войти через `/login`, перейти в `/admin`

> Механизм bootstrap срабатывает только один раз — пока в системе нет ни одного `super_admin`.

## Деплой

### Требования к инфраструктуре

- Сервер с Python 3.11+ и Node.js 18+ (или Docker)
- PostgreSQL 15 (managed или self-hosted)
- Redis 7
- SMTP или HTTP-почтовый сервис (Resend, SendGrid)
- HTTPS — обязателен (Let's Encrypt / nginx)

> **Платформы без исходящего SMTP (Railway, Render):** установите `SKIP_EMAIL_VERIFICATION=true`. В продакшне рекомендуется Resend или SendGrid через HTTPS.

```bash
# Задать продакшн-переменные
FRONTEND_URL=https://your-domain.com
BASE_VERIFICATION_URL=https://api.your-domain.com/api/v1/verify
ALLOWED_ORIGINS=https://your-domain.com

# Применить миграции (см. раздел ниже)
PYTHONPATH=. alembic upgrade head

# Запустить с несколькими воркерами
uvicorn app.main:app --host 0.0.0.0 --port 8200 --workers 4
```

Рекомендуется разместить за nginx с терминацией TLS и проксированием на `localhost:8200`.

### Frontend

```bash
cd frontend

# Задать переменные перед сборкой
# NEXT_PUBLIC_API_URL=https://api.your-domain.com
# API_URL=https://api.your-domain.com

npm install

```

### Раздельный деплой (Vercel + Railway)

Браузер не передаёт httpOnly-куки между доменами. Для решения этой проблемы фронтенд включает API-прокси (`/api/proxy/...`), который пересылает запросы на бэкенд с `Set-Cookie` на домене Vercel.

```env
# Vercel — переменные окружения
NEXT_PUBLIC_API_URL=/api/proxy   # клиент → прокси
API_URL=https://your-backend.up.railway.app  # SSR и middleware → напрямую
```

### Единый сервер

Если фронтенд и бэкенд на одном домене, прокси не нужен:

```env
NEXT_PUBLIC_API_URL=https://api.your-domain.com
API_URL=https://api.your-domain.com
```

### Docker Compose (all-in-one)

```bash
cd backend
cp example.env .env
# Заполнить .env
docker compose up -d --build
```

Поднимает FastAPI + PostgreSQL 15 + Redis. Миграции применяются автоматически через `entrypoint.sh`. Frontend запускается отдельно.

## Применение миграций

Миграции управляются через [Alembic](https://alembic.sqlalchemy.org). Все файлы находятся в `backend/migrations/versions/`.

### Локально

```bash
cd backend
source .venv/bin/activate

# Применить все новые миграции
PYTHONPATH=. alembic upgrade head

# Посмотреть текущую версию БД
PYTHONPATH=. alembic current

# Посмотреть историю миграций
PYTHONPATH=. alembic history --verbose

# Откатить последнюю миграцию
PYTHONPATH=. alembic downgrade -1
```

### На сервере после git push

После деплоя нового кода нужно применить миграции **до** перезапуска приложения. Порядок важен: приложение ожидает актуальную схему БД.

**Вариант 1 — вручную через SSH:**

```bash
# Подключиться к серверу
ssh user@your-server.com

# Перейти в директорию проекта
cd /opt/veritas/backend

# Активировать окружение
source .venv/bin/activate

# Применить миграции
PYTHONPATH=. alembic upgrade head

# Перезапустить приложение (зависит от того, как запущено)
sudo systemctl restart veritas-backend
# или:
pm2 restart veritas
# или (если Docker):
docker compose pull && docker compose up -d --build
```

**Вариант 2 — через Docker entrypoint (рекомендуется):**

В `backend/entrypoint.sh` уже настроено автоматическое применение миграций при старте контейнера:

```bash
#!/bin/sh
PYTHONPATH=. alembic upgrade head
exec uvicorn app.main:app --host 0.0.0.0 --port "${FASTAPI_PORT:-8200}"
```

При `docker compose up -d --build` миграции применяются автоматически перед запуском приложения.

**Вариант 3 — через Railway (CI/CD):**

В `railway.json` задайте команду старта:

```json
{
  "deploy": {
    "startCommand": "cd backend && PYTHONPATH=. alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port $PORT"
  }
}
```

При каждом деплое Railway выполнит эту команду — миграции применятся автоматически.

**Вариант 4 — GitHub Actions:**

```yaml
- name: Apply migrations
  run: |
    cd backend
    pip install -r requirements.txt
    PYTHONPATH=. alembic upgrade head
  env:
    DB_HOST: ${{ secrets.DB_HOST }}
    DB_USER: ${{ secrets.DB_USER }}
    DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
    DB_NAME: ${{ secrets.DB_NAME }}
```

### Создание новой миграции

```bash
cd backend
# Автогенерация на основе изменений в models.py
PYTHONPATH=. alembic revision --autogenerate -m "add_field_name"

# Проверить сгенерированный файл в migrations/versions/ перед применением
# Autogenerate не всегда точен: enum, constraints, индексы — проверять вручную

# Применить
PYTHONPATH=. alembic upgrade head
```

> **Важно.** Всегда проверяйте автосгенерированный файл миграции перед применением на продакшне. Особенно для: создания PostgreSQL enum-типов, изменения колонок с данными, добавления NOT NULL без default.

## Безопасность

| Механизм | Реализация |
|---|---|
| Пароли | Argon2id (устойчив к GPU-атакам) |
| JWT | httpOnly-куки, недоступны JavaScript |
| Refresh-токены | Redis, TTL 7 дней, немедленный revoke при logout |
| API-ключи | Хранятся как SHA-256-хэш; сам ключ показывается единожды |
| Email-токены | Одноразовые, TTL 30 минут |
| Подписи документов | HMAC/RSA; любое изменение данных в БД делает подпись недействительной |
| Журнал верификаций | PostgreSQL RULE запрещает UPDATE и DELETE на уровне СУБД |
| Rate limiting | Sliding window через Redis: 30 верификаций/мин с одного IP |
| Криптопровайдер | Абстракция: SHA-256/RSA → ГОСТ без изменения бизнес-логики |
