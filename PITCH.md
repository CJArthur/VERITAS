# VERITAS — Питч для жюри

> **Инфраструктурный слой доверия для образовательных документов**
> Diasoft Hackathon 2026

---

## 1. Проблема — реальная и измеримая

Каждый год в России продаются сотни тысяч поддельных дипломов. Врачи без медицинского образования проводят операции, финансисты не умеют считать риски, инженеры строят здания.

**Почему так происходит?** Проверить диплом кандидата сложно:

| Способ | Проблема |
|---|---|
| Позвонить в ВУЗ | «Перезвоните завтра» → ответ через дни |
| Официальный запрос | 5–30 рабочих дней, бюрократия |
| ФРДО (Рособрнадзор) | Нет открытого API для бизнеса; не охватывает частные учебные центры |
| Госуслуги | Нет B2B-интеграции |

**Итог: большинство работодателей вообще не проверяют.** Это системный сбой, который VERITAS устраняет.

## 2. Решение — 3 секунды вместо 30 дней

```
Организация загружает документ
       ↓
Система создаёт криптоотпечаток + цифровую подпись
       ↓
Выпускник получает QR-код
       ↓
Работодатель сканирует камерой
       ↓
Мгновенный результат в браузере
```

**Никаких приложений. Никакой регистрации для проверяющего. 3 секунды.**

Результат математически доказуем: подпись либо верна, либо нет. Это не «база данных которой можно доверять» — это криптография.

## 3. Правовая основа — почему это законно

VERITAS не заменяет государственную систему. Это **добровольный инструмент дополнительного доверия**, который работает в рамках действующего законодательства.

### Нормативная база

**ФЗ-273 «Об образовании в РФ», ст. 60** — образовательные организации несут ответственность за достоверность выданных документов. VERITAS предоставляет технический инструмент исполнения этой нормы в цифровом пространстве.

**ТК РФ, ст. 65** — работодатель вправе требовать документ об образовании и убедиться в его подлинности. VERITAS делает эту проверку мгновенной.

**ГК РФ, ст. 421** — свобода договора: стороны вправе использовать любые частные инструменты верификации. VERITAS — добровольная платформа для организаций, которые хотят предоставить проверяемые документы.

**ФЗ-63 «Об электронной подписи»** — электронная подпись обеспечивает юридически значимое подтверждение подлинности каждого документа, хранящегося в системе.

**ФЗ-152 «О персональных данных»** — студент сам даёт явное согласие на передачу данных конкретному работодателю через механизм TTL-ссылок. Без ссылки студента — доступа нет.

### Как регистрируются организации

Регистрация — двухэтапная. Каждая организация проходит **два независимых барьера**, которые невозможно обойти:

1. **Заявка:** организация предоставляет ОГРН, лицензию Рособрнадзора и аккредитацию.
2. **Автоматическая проверка:**
   - Контрольная сумма ОГРН по алгоритму ФНС — если ОГРН математически некорректен, одобрение **физически заблокировано** в системе. Администратор может только отклонить и связаться с организацией.
   - Поиск в ЕГРЮЛ через DaData API — официальное наименование, ИНН, статус (`ACTIVE` / `LIQUIDATED` / `BANKRUPT`)
   - Прямые ссылки на Рособрнадзор для проверки лицензии и аккредитации в один клик
   - Рекомендация системы: `✅ Одобрить` / `❌ Отклонить` / `⚠️ Проверить вручную` с обоснованием
3. **Финальное решение администратора:** человек видит готовый результат автопроверки и принимает решение. Ни одна фиктивная организация не проходит — математику не обмануть, а человека — не обойти.

Онлайн-школы и корпоративные учебные центры, выдающие сертификаты, регистрируются только по ОГРН — государственная лицензия не требуется. Тип документа (`certificate`, `professional_license`) указывается явно.

## 4. Что мы построили — рабочий MVP

### Три типа документов

| Тип | Кто использует | Пример |
|---|---|---|
| **Диплом** | ВУЗы | Диплом бакалавра МГУ |
| **Сертификат** | Онлайн-школы, корпоративные центры | Skillbox, Нетология |
| **Профессиональная лицензия** | Профессиональные организации | Медицинская лицензия, допуск СРО |

### Шесть киллер-фич

**1. Двойная верификация организаций — алгоритм + человек**

Когда ВУЗ или учебный центр регистрируется на платформе, система автоматически:
- Проверяет контрольную сумму ОГРН по алгоритму ФНС (математика, без сети). Если ОГРН не проходит — кнопка «Одобрить» заблокирована на уровне системы, невозможно нажать.
- Запрашивает DaData API (официальная обёртка над реестром ФНС) — получает юридическое наименование организации, ИНН и статус
- Формирует готовый результат с прямыми ссылками на реестры Рособрнадзора

Администратор открывает заявку и видит уже готовый светофор:
```
✅ ОГРН 1037700132712 — контрольная сумма верна
✅ ЕГРЮЛ: МГУ имени М.В. Ломоносова · ИНН 7701023558 · Статус: ACTIVE
📎 [ФНС ЕГРЮЛ ↗]  [Лицензии Рособрнадзор ↗]  [Аккредитации ↗]
🟢 Рекомендация: ОДОБРИТЬ
```

Один клик — и решение принято. Рутинного ручного поиска в реестрах нет. Но **финальное решение всегда за человеком** — именно это создаёт настоящее доверие.

**2. Криптографическая неизменяемость**

Каждый документ подписан приватным ключом организации. Даже при полной компрометации базы данных — изменить данные незаметно невозможно. Подпись либо совпадает, либо нет.

**3. Live Revoke Propagation**

Организация нажимает «Отозвать» → все открытые страницы верификации становятся красными в реальном времени, без перезагрузки. Студент не может «заморозить» страницу с действующим статусом. Это физически невозможно при нашей архитектуре.

*Демо: открываем зелёную страницу у работодателя → нажимаем Revoke в кабинете ВУЗа → страница краснеет прямо на глазах у жюри.*

**4. Privacy-first доступ (студент управляет)**

Без ссылки от студента — доступа нет. Студент создаёт TTL-ссылку:
- «Открыто персонально для Сбербанк HR»
- «До 15 апреля, максимум 3 просмотра»
- Видит кто, когда и сколько раз смотрел его документ
- Закрывает доступ одним нажатием

Это соответствует ФЗ-152: явное согласие на конкретную передачу ПДн.

**5. Аналитика для организаций**

ВУЗ видит в реальном времени:
- Сколько раз проверяли его выпускников и какие компании
- Топ специальностей по спросу у работодателей
- График верификаций за 30 дней с недельным сравнением

Это не просто красивый дашборд — это данные, которых у ВУЗов никогда не было. «Наших IT-выпускников проверяют Яндекс, Сбер и Газпром» — на дне карьеры это убедительный аргумент для абитуриентов.

**6. Иммутабельный аудит-журнал**

```sql
-- Физический запрет на уровне СУБД — не программное ограничение
CREATE RULE no_update_vlog AS ON UPDATE TO verification_logs DO INSTEAD NOTHING;
CREATE RULE no_delete_vlog AS ON DELETE TO verification_logs DO INSTEAD NOTHING;
```

Даже при взломе приложения или root-доступе к серверу — журнал нельзя изменить без прямого ALTER на PostgreSQL-кластере. Это audit-proof по определению.

## 5. Архитектура — почему она правильная

### Стек и обоснование

**FastAPI (Python) вместо Spring Boot (Java)**
- `async/await` из коробки — верификация это IO-bound задача (Redis → PostgreSQL → подпись)
- Pydantic v2 с Rust core — валидация данных быстрее большинства JVM-решений
- OpenAPI генерируется автоматически — API для работодателей готов из коробки
- Time-to-market: FastAPI быстрее в разработке на 30–40% для небольшой команды

**Next.js 14 App Router (SSR)**
- Страница `/v/[token]` — server component: работодатель видит результат мгновенно, без белого экрана и JS hydration
- Server Components снижают JS bundle в 3x
- Vercel деплой в один клик

**PostgreSQL + Redis**
- PostgreSQL: транзакционность, RULE для иммутабельности, сложные join-запросы для аналитики
- Redis: кэш верификаций (TTL 5 мин), мгновенный revoke-флаг, rate limiting (sliding window)

### CryptoProvider — замена алгоритма без рефакторинга

```python
# Одна строка в конфиге
CRYPTO_PROVIDER=sha256_rsa  # хакатон: SHA-256 + RSA-2048
CRYPTO_PROVIDER=gost        # прод: ГОСТ Р 34.11-2012 + ГОСТ Р 34.10-2012 (КриптоПро CSP)
```

Бизнес-логика не содержит ни одного импорта `hashlib` или `cryptography` — только интерфейс `CryptoProvider`. Переход на ФСБ-сертифицированные алгоритмы не требует рефакторинга.

### Безопасность данных

- ФИО хранится только в зашифрованном виде (AES-256-GCM на уровне приложения, в БД — BYTEA)
- Пароли — Argon2id (устойчив к GPU-атакам)
- JWT — httpOnly cookies (JavaScript не может прочитать)
- API-ключи — только SHA-256-хэш в БД, сам ключ показывается один раз
- Rate limiting: 30 верификаций/мин с одного IP через Redis

> ✅ ОГРН верен · ✅ ЕГРЮЛ: МГУ имени М.В. Ломоносова · ИНН 7701023558 · Статус: ACTIVE
> Нажимаем «Одобрить» — математика проверена автоматически, решение принял человек.

**Сцена 1: Быстрая проверка**
> Работодатель получает резюме с QR → сканирует телефоном → 3 секунды → зелёный экран.
> «Иванов Иван Иванович. МГУ. Прикладная математика. 2024. ДИПЛОМ ПОДЛИННЫЙ.»
> «Открыто для: Сбербанк HR · Действительно до: 20 апреля»

**Сцена 2: Live Revoke** *(главный вау-момент)*
> Оставляем зелёную страницу открытой на большом экране.
> Переходим в кабинет ВУЗа → «Отозвать» → причина «Решение учёного совета от 05.04.2026».

> «ДИПЛОМ НЕДЕЙСТВИТЕЛЕН. Аннулирован учебным заведением.»

**Сцена 3: Студент управляет доступом**
> Личный кабинет: «Кто смотрел мой диплом — Сбер, 3 раза».
> Создаём ссылку для нового работодателя: 7 дней, 2 просмотра, «для КРОК HR».
> После истечения — ссылка не работает. Студент в контроле.

**Сцена 4: Аналитика ВУЗа**
> Дашборд: «847 верификаций за месяц. Рост +34% к прошлой неделе».
> Топ проверяющих: Сбер (23), Яндекс (18), Газпром (11).
> «IT-специальности — самые востребованные по данным работодателей».

## 10. Ключевые тезисы для защиты

**«Почему не просто расширить ФРДО?»**
ФРДО охватывает только государственные дипломы. 40% рынка образования — частные учебные центры, онлайн-школы, корпоративные программы. VERITAS закрывает эту нишу и дополняет государственный реестр, а не конкурирует с ним.

**«Почему Python, а не Java/Go?»**
FastAPI async идентичен по производительности Go для IO-bound задач (а верификация — это Redis + PostgreSQL, не CPU). Pydantic v2 написан на Rust. OpenAPI из коробки. Команда пишет чистый, проверяемый код — это важнее для поддерживаемости.

**«Как масштабируется?»**
Верификация stateless: несколько инстанций FastAPI за балансировщиком, общий Redis, read-replicas PostgreSQL для верификации. Bottleneck — только БД, и он решается стандартными средствами.

**«Что насчёт безопасности данных?»**
152-ФЗ соблюдён: ПДн зашифрованы AES-256-GCM, согласие студента фиксируется при создании TTL-ссылки. Без явного согласия — работодатель не видит ПДн вообще. Архитектура соответствует УЗ-4.

**«Как проверить, что организация настоящая?»**
Двойной барьер, который невозможно обойти: математическая проверка ОГРН по алгоритму ФНС блокирует некорректные данные на уровне системы, а финальное решение принимает человек — администратор платформы, который при необходимости связывается с организацией напрямую через публичные контакты. Ни автоматика без человека, ни человек без автоматики — только вместе.

VERITAS — это не «удобная проверка диплома».

Это **инфраструктурный слой доверия** для образовательных документов: так же как HTTPS сделал веб защищённым, а S/MIME — email подписанным, VERITAS делает образовательные документы криптографически верифицируемыми, мгновенными и privacy-first.

**Документ без VERITAS — просто бумага. Документ в VERITAS — цифровой актив с математическим доказательством подлинности.**
